# Seasonal Business Simulator

Це repo-ready каркас під майбутню **детальну симуляцію бізнес-системи**, де кілька напрямів пов'язані між собою спільними ресурсами, сезонністю, оборотними коштами, технікою, персоналом та логістикою.

## Мета

Побудувати не "класичний бізнес-план", а **живу операційно-фінансову модель**, яка відповідає на питання:

- що дає найбільший cash flow;
- де виникають касові розриви;
- який напрям забирає спільний ресурс в іншого;
- коли варто входити в запас, а коли розвантажувати склад;
- яка техніка окупається швидше;
- які точки/напрями треба переносити, зупиняти або масштабувати.

## Поточні напрями

1. `01_apple_concentrate_network.md` — технічне яблуко / мобільні точки / перевалка / продаж на концентрат.
2. `02_firewood.md` — дрова як окремий сезонний напрям.
3. `03_technical_salt.md` — технічна сіль із сезонною закупкою і зимовим збутом.
4. `04_aggregates_building_materials.md` — пісок / щебінь / відсів / інші сипучі та будматеріали.
5. `05_transport_logistics.md` — логістика, фура, маніпулятор, внутрішні та зовнішні перевезення.

## Спільні документи

- `06_shared_resources_and_constraints.md` — спільні ресурси системи та конфлікти між напрямами.
- `07_seasonality_calendar.md` — календар сезонності.
- `08_legal_structure.md` — юридична модель.
- `09_apple_app_spec.md` — окремий документ по додатку для яблука.
- `10_simulation_architecture.md` — як це все перетворити в повноцінний симулятор.
- `11_data_dictionary.md` — перелік майбутніх полів для Excel / Python моделі.
- `12_discussion_backlog.md` — питання, які треба розібрати по черзі в подальших сесіях.

## Принцип роботи далі

1. Детально доповнюємо кожен напрям.
2. Формалізуємо всі вхідні параметри.
3. Визначаємо спільні обмеження.
4. Будуємо Excel-версію симулятора.
5. За потреби переносимо логіку в Python.

## Важливе правило

Усі твердження, які вже були зафіксовані раніше, залишаємо як **known facts**.  
Те, чого в контексті ще нема, фіксуємо окремо як **data to collect** або **working hypothesis**.

# Project Workflow and Memory Rules

## Мета цього файлу

Цей файл фіксує, **як саме ведеться проект `Business`**, як обробляється інформація, що вважається фактом, а що — гіпотезою, і який фінальний результат повинен бути побудований.

---

## 1. Межі контексту проекту

Уся інформація, яка надається в межах проекту `Business`, вважається **окремим проектним контекстом**.

### Правило ізоляції
- інформацію з `Business` не змішувати з іншими проектами без окремої команди;
- інформацію з інших проектів не переносити в `Business` автоматично;
- всі рішення, припущення, цифри та логіка по `Business` фіксуються окремо в цьому репозиторії.

---

## 2. Як подається нова інформація

Користувач поступово надає:

- загальний контекст;
- бачення моделі;
- плани закупок;
- плани продажів;
- логіку сезонності;
- варіанти техніки;
- варіанти юридичної моделі;
- варіанти кредитування;
- місця зберігання;
- цільові товари;
- канали збуту;
- інші стратегічні та операційні дані.

Після кожного великого блоку інформації ці дані повинні бути:

1. структуровані;
2. рознесені по відповідних `.md` файлах;
3. позначені як `known fact`, `working hypothesis`, `option`, `risk`, `data to collect` або `decision`.

---

## 3. Правила фіксації інформації

### Known fact
Те, що вже визначено і вважається реальною опорою для моделі.

### Working hypothesis
Те, що логічно виглядає правильним, але ще не підтверджене остаточно.

### Option
Один із варіантів рішення, який ще порівнюється з іншими.

### Risk
Фактор, який може зламати модель або сильно погіршити результат.

### Data to collect
Те, без чого модель ще не може бути доведена до точного розрахунку.

### Decision
Те, що вже обрано як робочий варіант і має впливати на наступні розрахунки.

---

## 4. Робочий цикл проекту

### Phase A — збір контексту
Користувач максимально детально пояснює:

- що хоче будувати;
- що саме купувати;
- що продавати;
- де працювати;
- коли заходити в сезон;
- як бачить логістику;
- які обмеження вже є;
- які рішення розглядаються.

### Phase B — структуризація
Уся інформація розкладається по документах проекту.

### Phase C — аналітика
По кожному напряму проводиться окремий детальний аналіз:

- логіка доходу;
- логіка витрат;
- потреба в оборотці;
- вузькі місця;
- системні конфлікти;
- сценарії;
- порядок масштабування.

### Phase D — інтеграція
Всі напрями зв'язуються в одну модель.

### Phase E — побудова симулятора
Створюється детальний Excel-файл, у якому враховуються:

- окремі бізнес-напрями;
- спільні ресурси;
- сезонність;
- ПДВ;
- ФОП;
- ТОВ;
- кредит або інше зовнішнє фінансування;
- cash flow;
- сценарії;
- точка беззбитковості;
- касові розриви;
- пріоритети росту.

---

## 5. Фінальний результат проекту

У фіналі проект повинен мати:

1. **детально оформлений GitHub-репозиторій** з усією логікою системи;
2. **максимально деталізований аналіз всієї бізнес-системи**;
3. **Excel-симулятор**, який моделює:
   - окремі напрями;
   - спільні ресурси;
   - юридичні режими;
   - податки;
   - кредитне навантаження;
   - сценарії розвитку;
   - ризики та обмеження;
4. за потреби — основу для наступного переносу в Python або окремий додаток.

---

## 6. Правило точності

Якщо даних бракує, вони **не вигадуються**.  
У такому випадку інформація маркується як:

- `unknown`;
- `requires validation`;
- `working assumption`.

---

## 7. Робочий принцип далі

Користувач дає новий блок контексту.  
Після цього:

- виконується структуризація;
- оновлюються відповідні файли;
- фіксуються рішення і невідомі;
- накопичується база для фінальної симуляції.
